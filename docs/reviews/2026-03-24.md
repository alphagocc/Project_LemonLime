## Findings (按严重级别)

### High

1. 未初始化指针 + 空指针调用风险（Memory Safety）  
   位置：src/core/contest.h:59, src/core/contest.cpp:234, src/core/contest.cpp:236  
   证据：`JudgingController *controller;` 未初始化，`stopJudgingSlot()` 直接调用 `QMetaObject::invokeMethod(controller, "stop")`。  
   风险：若在非评测态或异常流程触发停止，可能出现 UB / 崩溃。  
   建议：

- 将 `controller` 初始化为 `nullptr`。
- 在 `stopJudgingSlot()` 做空指针保护。
- 推荐 `QPointer<JudgingController>` 或 `std::unique_ptr<JudgingController>` 管理生命周期。

2. Contest 生命周期结束时存在任务对象泄漏窗口（Memory Safety）  
   位置：src/core/contest.h, src/core/contest.cpp, src/lemon.cpp:779  
   证据：`Contest` 持有 `QList<Task *> taskList` / `QMap<QString, Contestant *> contestantList`，但未定义析构清理逻辑；`LemonLime::closeAction()` 直接 `delete curContest`。  
   风险：`Task`（非 QObject parent 托管）可能在销毁 `Contest` 时泄漏。  
   建议：

- 为 `Contest` 增加析构，统一 `qDeleteAll(taskList)` / `qDeleteAll(contestantList)`（在 parent 关系明确后避免重复释放）。
- 中期将 owning 容器升级为 `std::vector<std::unique_ptr<Task>>` 与 `std::unordered_map<QString, std::unique_ptr<Contestant>>`（或 Qt 等效模式）。

3. 反序列化失败路径存在早退泄漏（Memory Safety + Dead Code 路径质量）  
   位置：src/core/contest.cpp:287-289, src/core/contest.cpp:299-301  
   证据：`new Task()` / `new Contestant()` 后若 `readFromJson(...) == -1` 直接 `return -1`，新建对象未释放。  
   风险：损坏数据输入会触发可累计泄漏。  
   建议：

- 用 `std::unique_ptr` 临时持有，解析成功后再 `release()` 入容器。
- 或先读取到局部对象，再统一提交。

4. 停止评测链路不完整，线程无法被即时中断（Correctness + Concurrency）  
   位置：src/core/taskjudger.cpp:472, src/core/taskjudger.cpp:492, src/core/taskjudger.cpp:503, src/core/taskjudger.cpp:544, src/core/taskjudger.h:87  
   证据：连接了 `TaskJudger::stopJudgingSignal -> JudgingThread::stopJudgingSlot`，但 `TaskJudger::stop()` 仅设置 `isJudging = false`，从未 `emit stopJudgingSignal`；同时大量 `thread->wait()` 阻塞。  
   风险：停止请求仅在 wait 返回后生效，无法及时打断长耗时 case。  
   建议：

- 在 `TaskJudger::stop()` 中补发 `stopJudgingSignal`。
- 将 `start()+wait()` 改为信号驱动或带超时轮询退出策略，避免硬阻塞。

### Medium

5. 排名构建逻辑在导出模块重复 4 次以上（DRY）  
   位置：src/component/exportutil/exportutil.cpp:265-272, src/component/exportutil/exportutil.cpp:603-610, src/component/exportutil/exportutil.cpp:700-707, src/component/exportutil/exportutil.cpp:786-787  
   证据：`sortList` 构造 + `std::sort` + `rankList` 计算在 HTML/SmallerHTML/CSV 等路径重复。  
   影响：规则一旦修改（并列排名、invalid 规则）需多点同步，容易漂移。  
   建议：抽取 `buildRankData(contest)` 返回排序列表和 rank 映射。

6. 编译器创建流程高度重复（DRY + Memory Safety）  
   位置：src/addcompilerwizard.cpp:426, src/addcompilerwizard.cpp:440, src/addcompilerwizard.cpp:483, src/addcompilerwizard.cpp:535, src/addcompilerwizard.cpp:549, src/addcompilerwizard.cpp:558, src/addcompilerwizard.cpp:573  
   证据：多分支重复 `auto *compiler = new Compiler;` + 若干 set + `compilerList.append(compiler)`。  
   影响：配置字段演进时漏改概率高。  
   建议：

- 抽取 `createCompiler(...)` / `appendBuiltinCompiler(...)`。
- 将 `compilerList` 改为 `QList<std::unique_ptr<Compiler>>`（或局部 unique_ptr + 转移所有权）。

7. API 设计导致不必要拷贝（Modernization/Optimization）  
   位置：src/core/contest.h:39, src/core/contest.cpp:71  
   证据：`QList<Contestant *> getContestantList() const { return contestantList.values(); }` 每次调用构造新列表。  
   影响：被多个视图/导出路径高频调用时产生额外分配。  
   建议：

- 提供 `const QMap<QString, Contestant *> &getContestants() const` 迭代接口。
- 或提供 visitor/迭代器回调，避免反复 materialize。

8. O(n^2) 重名检查可降为 O(n)（Modernization/Optimization）  
   位置：src/compilersettings.cpp:72-73  
   证据：循环内使用 `compilerNames.count(compilerNames[i]) > 1`。  
   建议：`QSet<QString>` 或 `QHash<QString, int>` 一次扫描完成计数。

9. 原始指针拥有关系分散，删除责任不集中（Memory Safety）  
   位置：src/base/settings.h:149, src/base/settings.h:166, src/core/contest.h:53  
   证据：`QList<Compiler *>`、`QList<ColorTheme *>`、`QList<Task *>` 多处手工 delete。  
   影响：异常路径/未来重构时易引入双删或泄漏。  
   建议：逐步将 owning 容器迁移到智能指针，并区分 owning 与 non-owning 视图。

## 四维度汇总

1. DRY Principle

- 主要重复集中在导出排名逻辑、编译器构建逻辑。
- 建议优先提取共享函数，再逐步引入策略对象减少分支复制。

2. Dead Code Elimination

- 存在未实现声明、注释残留、冗余 include。
- 建议建立 `-Wunused`/include-cleanup 的 CI 检查（可先 warning，不阻断）。

3. Modernization & Optimization

- 可显著优化点：避免 `getContestantList()` 高频拷贝、重名检查 O(n^2) -> O(n)。
- 建议在热点路径优先使用引用视图、范围 for、算法库封装公共流程。

4. Memory Safety & Raw Pointers

- 最高风险集中在 `Contest`/`TaskJudger` 生命周期管理与反序列化失败路径。
- 建议按“新代码先 RAII、旧代码渐进迁移”的策略推进，先覆盖 core 层 owning 容器。
