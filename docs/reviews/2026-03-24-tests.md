# Code Review: tests/ — 2026-03-24

审计范围：tests/CMakeLists.txt、tests/test1/CMakeLists.txt、tests/test1/main.cpp

## Findings (按严重级别)

### High

1. 全局工作目录被修改且未恢复，测试存在隐式耦合（Correctness）  
位置：tests/test1/main.cpp:106  
证据：`loadContest(...)` 内部执行 `QDir::setCurrent(contestDir);`，函数返回前后都没有恢复原目录。  
风险：后续测试依赖相对路径时会受到前序测试影响，出现顺序相关失败。  
建议：
- 引入 RAII 目录守卫（保存旧 cwd，析构恢复）。
- 或让被测代码支持显式路径参数，避免全局状态。

2. judgeAll 无超时保护，卡死会拖挂整个测试进程（Stability）  
位置：tests/test1/main.cpp:245-246  
证据：注释明确 `judgeAll()` 内部阻塞，随后直接调用 `contest->judgeAll();`，没有 watchdog/timeout。  
风险：评测线程异常时 CI 可能无限挂起。  
建议：
- 为集成测试加总超时（CTest `TIMEOUT` + 代码侧 watchdog）。
- 对关键等待点使用 `QTRY_*` 或超时 fail-fast。

### Medium

3. 失败路径上的手工 delete 不可靠（Memory Safety）  
位置：tests/test1/main.cpp:204, tests/test1/main.cpp:228, tests/test1/main.cpp:237, tests/test1/main.cpp:330-331  
证据：多处 `Settings *settings = createSettings(this);` 后依赖函数尾部 `delete settings;`。  
风险：若中途 `QVERIFY/QVERIFY2` 失败提前返回，析构代码不执行，产生测试过程泄漏。  
建议：
- 测试代码统一使用 `std::unique_ptr` / `QScopedPointer`。
- 把资源声明放在最小作用域，减少失败窗口。

4. createSettings 的 parent 形参是死参数（Dead Code）  
位置：tests/test1/main.cpp:32, tests/test1/main.cpp:87  
证据：`createSettings(QObject *parent = nullptr)` 内部 `Q_UNUSED(parent)`，对象创建为 `new Settings()` 未绑定 parent。  
建议：
- 删除形参，或实际使用 parent 建立 QObject 所有权。

5. 环境依赖过强：缺少编译器时直接 qFatal 终止（Portability）  
位置：tests/test1/main.cpp:53, tests/test1/main.cpp:73  
证据：`g++`/`python` 未找到则 `qFatal(...)`。  
风险：在精简 runner 或交叉平台环境中，测试直接硬失败，无法区分“环境缺失”与“逻辑回归”。  
建议：
- 改为 `QSKIP`（标记环境不满足）。
- 结合 CMake 选项区分“必须集成测试”和“可选环境测试”。

6. 目录拷贝工具缺少返回值校验（Robustness）  
位置：tests/test1/main.cpp:147, tests/test1/main.cpp:151, tests/test1/main.cpp:158  
证据：`QDir().mkpath(...)` 与 `QFile::copy(...)` 返回值未校验。  
风险：拷贝失败时测试仍继续，后续失败信息偏离根因。  
建议：
- 对 `mkpath/copy` 使用 `QVERIFY2`，在初始化阶段尽早失败并给出路径上下文。

7. 断言代码重复且可读性下降（DRY）  
位置：tests/test1/main.cpp:305-327  
证据：user1 六个 case 使用重复模板断言；字符串拼接与错误信息模板大量重复。  
建议：
- 使用数据驱动测试表（`QTest::addColumn/newRow`）。
- 抽取 `assertCaseResult(contestant, taskIdx, caseIdx, expectedSet)`。

8. 测试职责混合，定位回归原因成本高（Maintainability）  
位置：tests/test1/main.cpp:233-234, tests/test1/main.cpp:236  
证据：注释说明“Test 3&4 合并为一个 slot”，单测试同时覆盖 judge 与结果校验。  
风险：失败时难区分是“评测流程失败”还是“断言矩阵失败”。  
建议：
- 保持共享夹具，但拆分为更细的测试点（流程完成、用户2结果、用户1结果）。

### Low

9. 可疑冗余 include（Dead Code/清理）  
位置：tests/test1/main.cpp:23, tests/test1/main.cpp:25  
证据：`#include <QProcess>`、`#include <QTextStream>` 在当前文件未直接使用。  
建议：移除未使用 include，减少编译负担与头文件噪音。

10. CMake 测试命名可扩展性一般（Scalability）  
位置：tests/CMakeLists.txt:1, tests/test1/CMakeLists.txt:47  
证据：当前仅 `test1`，后续扩展多测试集时命名语义较弱。  
建议：采用 `contest.integration.basic` 之类层级化命名，便于 `ctest -R` 精准筛选。

## 四维度汇总

1. DRY Principle  
- 主要重复点在判题结果断言模板，可通过数据驱动测试压缩样板代码。

2. Dead Code Elimination  
- 已识别死参数与疑似冗余 include，可快速清理。

3. Modernization & Optimization  
- 建议将资源管理改为 RAII，失败路径不再依赖手工 delete。

4. Memory Safety & Raw Pointers  
- 测试代码中裸指针删除路径受断言短路影响，建议全面改为智能指针。
